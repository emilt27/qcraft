# Дизайн: `BinaryOp::Power` (`**`) та `BinaryOp::BitwiseXor` (`^`)

## Проблема

Модель amsdal `Combinable` виставляє дев'ять інфіксних операторів. Сім із них
відображаються на `qcraft_core::ast::expr::BinaryOp`; два — ні:

- `POW` = `**` (піднесення до степеня)
- `XOR` = `^` (побітовий XOR)

Наразі гілка `"Combined"` у PyO3-біндінгу кидає `Unknown combined expression
operator` для цих операторів, тож `Value(2) ** Value(3)` і `a ^ b` дають жорстку
помилку на етапі екстракції. До-qcraft генератор підтримував `**` (рендерив
`POWER(l, r)`); це регресія можливостей, яку треба закрити **саме в qcraft**,
використовуючи нативний вбудований механізм кожного діалекту там, де він є.

| Оператор     | Postgres (нативно)          | SQLite (нативного оператора немає)   |
|--------------|-----------------------------|--------------------------------------|
| `Power`      | `left ^ right` (у PG `^` = піднесення до степеня) | `power(left, right)` (математична вбудована функція) |
| `BitwiseXor` | `left # right` (у PG `#` = побітовий XOR)         | `(left | right) - (left & right)`    |

- У PG `^` — це **піднесення до степеня**, а НЕ XOR — саме тому POW→`^` і XOR→`#` на PG.
- SQLite `power()` потребує математичного розширення (увімкнене за замовчуванням у
  SQLite ≥ 3.35). Якщо збірка його не має, `power()` падає в рантаймі — це прийнятно;
  емулювати не треба.
- SQLite XOR: `(a | b) - (a & b)` дорівнює `a ^ b` для всіх цілих (біти `&` є підмножиною
  бітів `|`, тож віднімання ніколи не «позичає»).

## Керівний інваріант (чому форма рішення визначена наперед)

**AST у qcraft — діалект-незалежний; діалект-специфічні розбіжності живуть лише в рендерері.**

Наслідки, що обмежують дизайн:

- Power/XOR **мусять** бути вузлами AST, кожен з яких рендериться по-своєму в кожному
  діалекті. Не можна «зашити» `power(...)` у білдер AST / біндінг, бо одне дерево
  рендериться *будь-яким* із діалектів.
- Наявна точка розширення `BinaryOp::Custom(Box<dyn CustomBinaryOp>)` **не підходить**:
  вона лише інфіксна (`render_custom_binary_op` просто пише рядок оператора), а SQLite
  відкидає всі кастомні бінарні оператори. Power/XOR — це стандартні портативні
  оператори, а не нішеві розширення на кшталт pgvector — вони заслуговують на
  first-class варіанти.

Отже: два нові first-class варіанти `BinaryOp`.

- Право-асоціативність PG `^` / те, що він зв'язує тісніше за `*`, — **не турбота qcraft**.

  будує саме таке дерево.
- Єдине самогрупування, яке належить qcraft, — це обгортання SQLite-композиту XOR так, щоб
  він був безпечним усередині будь-якого батька (і кожного операнда — заради пріоритетів).
- **Ніякої таблиці пріоритетів не додається і не потрібно.**

## Обсяг рішення (обидва режими параметрів)

Щоб SQLite XOR працював **повністю**, включно з незв'язаними `Expr::Param` у
`executemany`/batch, вводимо **нумеровані плейсхолдери** (`?N`). Це дозволяє відрендерити
композит `(l|r)-(l&r)`, посилаючись на кожен логічний операнд кілька разів, але
зв'язуючи його **один раз** на рядок.

Спайк підтвердив підтримку драйвером (Python `sqlite3` 3.45.3,
`scratchpad/spike_numbered.py`): `?N` reuse працює в `execute()` і `executemany()`; драйвер
рахує потрібну кількість зв'язувань як **найбільший індекс N**, значення зв'язуються **за
номером**, невідповідність кількості → гучний `ProgrammingError`.

## Дизайн

### 1. Core AST — `crates/qcraft-core/src/ast/expr.rs`

Додати два варіанти до `enum BinaryOp` (Power поруч з арифметичними операторами,
BitwiseXor поруч з іншими побітовими — для читабельності; порядок не впливає на поведінку):

```rust
pub enum BinaryOp {
    Add, Sub, Mul, Div, Mod,
    Power,       // ** піднесення до степеня
    BitwiseAnd, BitwiseOr,
    BitwiseXor,  // ^ побітовий XOR
    ShiftLeft, ShiftRight, Concat,
    Custom(Box<dyn CustomBinaryOp>),
}
```

Додати перевикористовуваний предикат на `Expr` (природне місце — core; тримає SQLite-рендерер чистим):

```rust
impl Expr {
    /// True, якщо дерево виразу містить незв'язаний плейсхолдер `Expr::Param`.
    /// Захист від псування позиційного зв'язування при double-render (ненумерований режим).
    /// Рекурсивно покриває під-виразні варіанти: Binary/Unary/Func/Cast/Case/Tuple/
    /// Aggregate/Window/JsonArray/JsonObject/JsonAgg/StringAgg/JsonPathText/Collate.
    /// У QueryStmt-підзапити НЕ заходить — вони й так відхиляються `contains_subquery`.
    pub fn contains_unbound_param(&self) -> bool { /* рекурсивний обхід */ }

    /// True, якщо дерево виразу містить підзапит (`Exists`/`SubQuery`/`ArraySubQuery`),
    /// у т.ч. вкладений (напр. `(SELECT ...) + 1`). Захист від подвійного ВИКОНАННЯ
    /// у SQLite-композиті XOR. Рекурсивно покриває ті самі під-виразні варіанти.
    pub fn contains_subquery(&self) -> bool { /* рекурсивний обхід */ }
}
```

### 2. Core render — `crates/qcraft-core/src/render/ctx.rs`

**2а. Новий стиль параметрів** — нумерований QMark:

```rust
pub enum ParamStyle {
    Dollar,        // $1, $2  (Postgres)
    QMark,         // ?       (SQLite/MySQL, як зараз)
    QMarkNumbered, // ?1, ?2  (SQLite, для reuse у composite/executemany)
    Percent,       // %s      (psycopg)
}
```

У `placeholder()` і `raw_with_params()` додати гілку `QMarkNumbered => "?{param_index}"`
(лічильник `param_index` уже існує — Dollar друкує `$N`). У нумерованому режимі **всі**
плейсхолдери стейтменту нумеровані (не змішувати з анонімними `?` — зі спайку).

**2б. Примітив `capture` для повторного використання фрагмента** — рендерить піддерево
один раз, повертає його SQL-текст, прибирає текст з буфера, але **лишає просунутими**
`param_index`/`params`. Потребує `use crate::error::RenderResult;` на початку `ctx.rs`
(зараз не імпортовано; `error.rs` автономний — циклу немає):

```rust
/// Рендерить фрагмент через `f`, повертає його SQL-текст (без провідних/кінцевих
/// пробілів) і прибирає з буфера, лишаючи param_index / params просунутими рівно
/// на один рендер операнда. Текст можна вставити кілька разів: у нумерованому
/// режимі `?N` посилаються на ті самі вже зареєстровані параметри.
pub fn capture<F>(&mut self, f: F) -> RenderResult<String>
where
    F: FnOnce(&mut RenderCtx) -> RenderResult<()>,
{
    let start = self.sql.len();
    f(self)?;
    // ТРІМ обов'язковий: space_if_needed міг додати провідний пробіл залежно від
    // контексту перед виразом; без тріму фрагмент став би контекстозалежним і
    // точні рядкові тести попливли б.
    let frag = self.sql[start..].trim().to_string();
    self.sql.truncate(start); // start — валідна межа (попередня довжина)
    Ok(frag)
}
```

**2в. Плумбінг param_style у `SqliteRenderer`** — зараз `pub struct SqliteRenderer;` це
**unit-структура без поля**, і всі 4 точки входу (`render_schema_stmt`,
`render_transaction_stmt`, `render_mutation_stmt`, `render_query_stmt`) хардкодять
`RenderCtx::new(ParamStyle::QMark)`. Щоб `QMarkNumbered` був досяжний:

```rust
pub struct SqliteRenderer {
    param_style: ParamStyle, // default QMark
}
impl SqliteRenderer {
    pub fn new() -> Self { Self { param_style: ParamStyle::QMark } }
    pub fn with_param_style(mut self, style: ParamStyle) -> Self {
        self.param_style = style; self
    }
}
```

Кожен `RenderCtx::new(ParamStyle::QMark)` замінити на `RenderCtx::new(self.param_style)`
(зберігаючи `.with_parameterize(true)` там, де воно є). `impl Default` лишається. Так
`amsdal-glue` вмикає режим через `SqliteRenderer::new().with_param_style(ParamStyle::QMarkNumbered)`,
а існуючі споживачі на `::new()` лишаються на `QMark` (backward-compatible).

### 3. Рендерер Postgres — `crates/qcraft-postgres/src/lib.rs` (~рядок 894)

Чистий інфікс. **Ніякого структурного розгалуження.** Додати два записи в match рядків операторів:

```rust
BinaryOp::Power      => "^",
BinaryOp::BitwiseXor => "#",
```

Обробка `mod_op` `%`/`%%` для стилю параметрів не зачіпається. Це вся зміна для PG.

**Жодних захистів на PG не потрібно.** Обидва оператори нативні й інфіксні — операнди
рендеряться **один раз**, подвоєння немає. Тож підзапити, незв'язані `Param`, волатильні
вирази як операнди `Power`/`BitwiseXor` на Postgres працюють без обмежень. Захисти нижче —
виключно SQLite-специфічні (наслідок емуляції XOR композитом).

### 4. Рендерер SQLite — `crates/qcraft-sqlite/src/lib.rs` (~рядок 469)

Поточна гілка пре-рендерить `left` до того, як зматчити оператор — це прибираємо; `left`
рендериться лише у fallback (інфікс/кастом). Дві неінфіксні гілки:

**4а. `Power` → `power(l, r)`** — операнди рендеряться **один раз**, тож працює однаково в
обох режимах параметрів, без спецлогіки (за зразком `Expr::Func`):

```rust
BinaryOp::Power => {
    ctx.keyword("power").write("(");
    self.render_expr(left, ctx)?;
    ctx.comma();
    self.render_expr(right, ctx)?;
    ctx.paren_close();
    Ok(())
}
```

**4б. `BitwiseXor`** — композит дублює операнд, тож поведінка залежить від режиму параметрів.

**Канонічна форма виходу (обидва режими, однакова):**
```
(((L) | (R)) - ((L) & (R)))
```
Дужки **обов'язкові й усі потрібні** (пріоритети SQLite: `-` рівень 4 тісніше за `|`/`&`
рівень 5):
- **зовнішні** `(...)` — бо qcraft НЕ авто-дужкує; без них батько `x * (a^b)` дав би
  `(x*(a|b)) - (a&b)`;
- **OR-група** `((L)|(R))` — без неї `-` перетягнув би: `(L)|((R)-((L)&(R)))`;
- **AND-група** `((L)&(R))` — правий операнд `-`;
- **кожен операнд** `(L)`, `(R)` — щоб складений операнд (`x<<y`, `a|b` тощо) не змішався
  з `|`/`&` на тому самому рівні.

**Захисти (SQLite-специфічні, бо композит дублює операнд):**
1. **Підзапит-операнд → завжди помилка (обидва режими).** Композит вставляє текст операнда
   двічі, тож підзапит **виконався б двічі** (продуктивність + недетермінізм). `?N` це НЕ
   рятує: capture-splice копіює *текст*, а текст містить `SELECT ...` → рушій проганяє його
   двічі. Тож `left.contains_subquery() || right.contains_subquery()` →
   `RenderError::unsupported`. XOR підзапитів безглуздий, тож блокуємо чесно.
2. **Незв'язаний `Param` у ненумерованому режимі → помилка.** Double-render подвоїв би
   позиційні плейсхолдери. У `QMarkNumbered` capture-splice це вирішує (параметр зв'язаний
   раз), тож там дозволено.

Режими рендеру:
- **`QMarkNumbered`:** render-once, capture-splice. Операнд рендериться один раз через
  `ctx.capture` (фрагмент трімлений), текст вставляється двічі; `?N` посилаються на ті самі
  зареєстровані параметри → кожен логічний параметр зв'язаний **один раз**. Довільно складні
  **детерміновані** операнди, включно з незв'язаними `Param`.
- **`QMark` / `Percent`:** capture-splice неможливий (анонімний `?` не перевикористати) →
  безпечний double-render для полів/`Value`.

```rust
BinaryOp::BitwiseXor => {
    // Захист 1: підзапит-операнд виконався б двічі — блокуємо в будь-якому режимі.
    if left.contains_subquery() || right.contains_subquery() {
        return Err(RenderError::unsupported(
            "BitwiseXor",
            "SQLite XOR емулюється композитом, що дублює операнд; операнд-підзапит \
             виконався б двічі — не підтримується (використайте Postgres або перепишіть)",
        ));
    }
    if ctx.param_style() == ParamStyle::QMarkNumbered {
        let l = ctx.capture(|c| self.render_expr(left, c))?;   // трімлений фрагмент
        let r = ctx.capture(|c| self.render_expr(right, c))?;
        ctx.paren_open();                                              // (
        ctx.write("((").write(&l).write(") | (").write(&r).write("))"); // ((l) | (r))
        ctx.write(" - ");
        ctx.write("((").write(&l).write(") & (").write(&r).write("))"); // ((l) & (r))
        ctx.paren_close();                                             // )
        Ok(())
    } else {
        // Захист 2: незв'язаний Param зіпсував би позиційне зв'язування при double-render.
        if left.contains_unbound_param() || right.contains_unbound_param() {
            return Err(RenderError::unsupported(
                "BitwiseXor",
                "SQLite XOR у ненумерованому режимі дублює операнди; незв'язаний Param \
                 зіпсував би позиційне зв'язування — увімкніть ParamStyle::QMarkNumbered",
            ));
        }
        ctx.paren_open();                                             // (
        ctx.write("((");  self.render_expr(left, ctx)?;  ctx.write(") | (");
        self.render_expr(right, ctx)?;  ctx.write("))");             // ((left) | (right))
        ctx.write(" - ");
        ctx.write("((");  self.render_expr(left, ctx)?;  ctx.write(") & (");
        self.render_expr(right, ctx)?;  ctx.write("))");             // ((left) & (right))
        ctx.paren_close();                                           // )
        Ok(())
    }
}
```

Обидві гілки дають однакову канонічну форму. Потрібен геттер `ctx.param_style()`
(поле `param_style` приватне, геттера зараз немає).

> **Волатильні скалярні функції** (`random()` тощо) як операнд теж виконаються двічі, але
> їх не виявити структурно без таблиці волатильності — не блокуємо (нереалістично для
> цілочисельного XOR); детермінізм операнда лишається відповідальністю викликача. Підзапити,
> на відміну, виявляються структурно → блокуються (Захист 1).

### 5. Дужки / пріоритети — таблиця не додається

Групування задає викликач через `Expr::Tuple`. Самогрупування qcraft — лише всередині
SQLite-композиту XOR (зовнішні дужки + дужки навколо кожного операнда).

### 6. Складні операнди, вкладення, ефективність

- **`(x+y) ^ z`** (нумерований): `capture(x+y)` → `?1 + ?2`, `capture(z)` → `?3`, результат
  `(((?1 + ?2)) | ((?3))) - (((?1 + ?2)) & ((?3)))`; рядок дає `(x, y, z)` — 3 значення.
  (Операнд-дужки + групові дужки дають кілька шарів — надлишок косметичний, коректність
  понад усе.)
- **Вкладення** `(a ^ b) ^ c` працює рекурсивно: внутрішній XOR captured як фрагмент,
  зовнішній компонує з нього; параметри лишаються `[a, b, c]`.
- **Ефективність:** один обхід дерева на операнд (другий екземпляр — копія *тексту*);
  параметри лінійні за кількістю логічних параметрів. `capture` робить одну алокацію
  фрагмента на операнд (2 на XOR).
- **Прийнятий трейд-оф:** текст SQL подвоюється на кожен рівень вкладення XOR (операнд
  з'являється 2× на рівень) — O(2^depth) за розміром тексту, хоча **параметри лишаються
  лінійними**. Неминуче в SQLite (немає посилання на обчислений проміжний результат поза
  CTE). Для реальної глибини (2–3) — дрібниця; приймаємо як є, без штучного ліміту.
- **Обмеження коректності:** підзапит-операнд відхиляється (Захист 1, обидва режими);
  волатильна скалярна функція виконалась би двічі — не блокується, відповідальність
  викликача (див. 4б). На Postgres обмежень немає (нативний інфікс, без подвоєння).
- **Мапінг `?N`↔`params[i]`** тримається, лише коли плейсхолдери в стейтменті однорідні
  (усі bound *або* усі unbound) — це pre-existing інваріант нумерованих стилів (той самий
  для Dollar `$N`), а не новий ризик; змішування bound+unbound в одному стейтменті лишається
  поза контрактом.

## Тестування (TDD — кожен тест падає на немодифікованому коді, проходить після зміни)

**Postgres:**
- Power: `2 ** 3` → `2 ^ 3`; з `FieldRef`/`Value`.
- XOR: `a ^ b` → `a # b`.
- Достовірне вкладення (без авто-дужок): `Tuple([a + b]) ** 2` → `(a + b) ^ 2`.
- **XOR/Power із підзапит-операндом працює** (нативний інфікс, без подвоєння) —
  `subq ^ b` → `(SELECT ...) # b`, без помилки. Доводить, що захисти SQLite-специфічні.
- Регресія `%`/`%%` при `ParamStyle::Percent`.

**SQLite — `QMark` (як зараз):**
- Power: `2 ** 3` → `power(2, 3)`; `power((a + b), 2)` для `Tuple([a+b]) ** 2`.
- XOR лише з полями/літералами: `a ^ b` → `(((a) | (b)) - ((a) & (b)))` (канонічна форма);
  `Value` пушиться двічі, порядок параметрів коректний.
- **XOR всередині батьківського виразу** (ловить Баг 2): `1 + (a ^ b)` →
  `1 + (((a) | (b)) - ((a) & (b)))` — зовнішні дужки ізолюють композит від `+`.
- **XOR із підзапит-операндом → `RenderError::unsupported`** (Захист 1; вкладений теж:
  `((SELECT ...) + 1) ^ b`). Падає до захисту, проходить після.
- XOR із незв'язаним `Param` → `RenderError::unsupported` (Захист 2; падає до захисту).

**SQLite — `QMarkNumbered` (нова функціональність):**
- Базові плейсхолдери: звичайний запит друкує `?1, ?2, ...` у порядку.
- XOR простий: `a ^ b` → `(((?1) | (?2)) - ((?1) & (?2)))`, значення `[a, b]` (2, не 4).
- XOR складний: `(x+y) ^ z` → канонічна форма з `?1,?2,?3`, **3** логічні параметри.
- XOR вкладений: `(a ^ b) ^ c` → параметри `[a, b, c]`, коректний XOR-результат.
- **Тест пріоритетів через реальний SQLite** (ловить Баг 1 і Баг 2): виконати
  `SELECT <xor>` І `SELECT k + <xor>` через `rusqlite`, звірити з `a ^ b` (та `k + (a^b)`)
  у Rust. Буквально обчислює значення → неправильні дужки провалять тест.
- **Семантичний тест reuse** (executemany-аналог): виконати згенерований SQL із набором
  рядків, звірити з `a ^ b` — доводить, що reuse не лише текстово коректний, а й дає
  правильний XOR при одному зв'язуванні на логічний параметр.

## Вичерпність (exhaustiveness)

Додавання варіантів змусить компілятор підсвітити будь-який match на `BinaryOp` без
wildcard. Підтверджено, що існує лише два місця з match (`qcraft-postgres/src/lib.rs`,
`qcraft-sqlite/src/lib.rs`); обидва явно перелічують варіанти, а тоді мають fallback — додати
два нові варіанти до кожного явного переліку. Аналогічно match на `ParamStyle` у
`ctx.rs` (`placeholder`, `raw_with_params`) отримає нову гілку.

## Поза межами (out of scope)

- **Downstream-біндінг amsdal-glue** — окремий репозиторій, окреме продовження після
  публікації qcraft. Потрібно: (а) гілка `"Combined"` у `extract.rs` мапить
  `"**" => BinaryOp::Power`, `"^" => BinaryOp::BitwiseXor` (прибрати з гілки помилки);
  (б) SQLite-конекшн вмикає `ParamStyle::QMarkNumbered` і в batch-шляху передає одне
  значення на логічний параметр (рядок лишається `(a, b)`, не `(a, b, a, b)`);
  (в) golden-тести. Закриває finding I1 рев'ю міграції sql-rust.
- Загальний фреймворк «оператор, що рендериться як функція» — тут потрібен лише Power/SQLite;
  лишаємо це точковою гілкою (YAGNI).
- Емуляція PG-піднесення до степеня на збірках SQLite без математичного розширення —
  нехай `power()` віддає нативну помилку.
- Штучний ліміт глибини вкладеного XOR — не вводимо (трейд-оф тексту прийнятий).
